Getting Started

Welcome to the official Engram infrastructure documentation. This guide gives you everything you need to start operating autonomous workloads securely within minutes.

CRITICAL SECURITY NOTICE

All Engram safety modules are designed for strict default-deny execution boundaries. Confirm policy triggers, gating, and local host controls are active before starting your agent runs.

Installation

Install the SDK via pip:

pip install engram-sdk

Minimal 5-Minute Example

Register your first agent and authorize an action to see the core control loop in action. One complete flow from start to finish:

import engram # 1. Initialize the core core = engram.init(api_key="sk-...", env="local") # 2. Register an agent agent = core.identity.register( role="DevOps Bot", permissions=["deploy_server", "read_logs"] ) # 3. Authorize an action result = core.enforcement.authorize( agent_id=agent.id, action={"type": "deploy_server", "target": "production"} ) if result.is_approved: print("Action permitted: Executing...") else: print(f"Action denied. Reason: {result.rejection_reason}")

If developers cannot run this in 5 minutes, we have failed. This snippet proves our simple mental overhead while protecting operations strictly.

Core Concepts

Before diving into APIs, understand the Engram mental model. Our system abstracts complex policy rules into simple, enforceable layers.

  • What is an Agent? Independent workloads operating under roles. Agents perform actions but have zero inherent authority until verified.
  • What is Identity? Your agents are treated as users. They authenticate, verify their origins, and adopt identities that determine their clearance level.
  • What is Policy Enforcement? The execution barrier. Before execution, actions are mapped against allowed logic matrices.
  • What is Economic Autonomy? Built-in constraints (budgets, usage limits). Agents cannot spend beyond algorithmic bounds, physically ensuring no runaway costs.
  • What is Delegation? Scoped authority transfer. An agent can spin up a sub-agent with a strict subset of its own permissions.
  • What is Simulation? Testing actions against live policies without actually executing them, calculating risk scores securely.
  • What is Governance? State transitions via proposals. To update an agent's core capability, proposals go through programmable consensus.

Your agent logic communicates locally with the SDK, which invokes the core infrastructure. The core routes through strictly segregated packages (like enforcement rules) before confirming action states to the final server output.

SDK Reference

The public methods mapping agent actions into the framework.

authorize(agent_id: str, action: dict) → AuthorizationResult

Submits a requested action to the enforcement layers for immediate deterministic risk evaluation.

Parameters
  • agent_id - The registered identity of the executing agent.
  • action - Dict containing type and specific target payload.
Return Values

Returns an AuthorizationResult with a boolean flag is_approved, and a strict rejection_reason string if false.

Error Cases

Throws IdentityMismatchError if ID is invalid. Throws TimeoutError if internal governance takes too long simulating the scope.

Module Documentation

Each foundational package isolates a specific vector of agent control.

Agent Guard

Agent Guard verifies identity before any agent-to-agent action is accepted, so workflows only continue when both sides can prove they are trusted instances.

  • Checks each agent's cryptographic identity token against the Identity System registry.
  • Rejects requests when signatures, role claims, or trust scope do not match policy.
  • Maintains a verification trail for audits across multi-agent sessions and escalations.

Identity System

  • Register agent dynamically.
  • Assign role and structural hierarchy constraints.
  • Authority mapping for multi-tenant environments.

Enforcement Layer

  • Define policies in readable, strict formats.
  • Rule evaluation lifecycle guarantees no race conditions.
  • Rejection handling for immediate operation halts.

Economic Autonomy

  • Create budgets at the agent or cluster level.
  • Set constraints that strictly block executions when thresholds meet.
  • Dynamic adjustments via governance approval only.

Scoring Module

  • Risk scoring based on historical anomaly vectors.
  • Threshold triggers to push manual review.
  • Decision outputs mapping mathematically to safety constraints.

Simulation Layer

  • Calculates potential outcomes in a shadow-state memory graph.
  • Use case: High-risk database operations or financial triggers.
  • Performance considerations: Expect ~20ms latency overhead during complex simulations.

Advanced Usage

For serious practitioners seeking to override base mechanisms or implement complex integrations.

  • Custom policy engines: Inject your own evaluation AST matching your server's security language.
  • Extending modules: Override BaseModule interface to connect natively to internal messaging buses.
  • Writing custom scoring plugins: Import our engram.plugins spec to connect third-party anomaly ML models.
  • Integrating with external databases: Set persistence layers outside local SQLite, moving toward Postgres mapping.
  • Running locally vs production: Bypass heavy verifications locally while forcing strict signatures in production via ENGRAM_ENV=production.

Governance & Self-Improvement

Our true differentiator: Programmable organizational changes via clear states.

  • Proposal lifecycle: Submit capability updates to agents, moving into an "Under Review" state.
  • Voting / approval logic: Set custom multi-sig rules relying on human-in-the-loop sign-off.
  • Versioning: Every agent iteration is hashed.
  • Rollback mechanism: An action that violated unknown parameters instantly triggers a rollback script returning to a safe agent hash state.

Deployment Guide

We do not lock you into cloud environments. Keep operational logic local where possible.

  • Local deployment: Run the background processes purely as daemons pointing to your system sockets.
  • Self-hosted server setup: Provided Dockerfiles mapping your control states securely behind standard API gateways.
  • Running as a background service: Seamless systemctl mappings or raw background Python hooks.
  • Production configuration notes: Secure the API keys entirely out of memory, rely strictly on environment variables or vaults.

Security Model

Trust but strictly verify. Safety isn't an afterthought; it is our foundation.

  • Deterministic enforcement vs LLM guardrails: We rely on zero LLM "vibe checks" for security. If a policy fails a deterministic logical rule, it is halted instantly.
  • Layered validation: Identity > Economic > Policy > Execution boundaries.
  • Audit logs: Read-only cryptographic hashing of every decision tree for audit compliance.
  • Failure modes: Fail-closed by default. If the core connects but enforcement timeouts, execution is rejected.
  • Threat model: We defend against injected execution requests, rogue credential misuse, and exponential recursive loops.

Error Handling & Debugging

Common failure cases and operational fixes:

  • Policy rejection: Check if your agent's role permits the requested target array payload. Output verbose logs via core.debug_logs().
  • Budget exceeded: Ensure the sub-agent wasn't cloned outside the parent budget scope block.
  • Identity mismatch: Verify your keys correlate to the active engram.env definition block.
  • Delegation scope violation: Standard fail-safe. Review your delegation mapping tree—agents cannot grant permissions they do not own.
